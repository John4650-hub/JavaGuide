In many cases, for work reasons, we need to learn a technology quickly and apply it in projects. Or, in order to respond to interview needs, we need to quickly master a technology that we have not been in touch with before.

As a purely self-taught programmer, the article briefly shares its views on how to learn a technology quickly.

When learning any technology, it is important to know what problem it is meant to solve. Before in-depth learning about the technology, it must be based on a holistic perspective, reflecting on the modules it is made up of, the functions it provides, and the advantages it must have.

For example, when we study Spring, we can learn from Spring's official documentation what the latest technological developments are, what modules Spring contains, and how Spring can help us.

![Image](https://oss.javaguide.cn/github/javaguie/system-design/web-real-time-message-push/202056110341227.png)

And for example, when I study the news queue, I'm going to find out how the news queue usually works in the system and how it helps us solve problems. There are many types of news queues, and I compare them with those I've learned. When I study RocketMQ, I compare it with the first section of ActiveMQ I've ever studied, and I think about how RocketMQ has been upgraded in relation to ActiveMQ, how it has solved the pains of ActiveMQ, and where it is similar and where it is different.

**The fastest way to learn a technology is to connect it with the technologies that you have learned before into a network.**

Then I suggest you look at the curriculum of official documents, run the relevant demos, and do some small projects.

However, official documents are usually available in English and are often only available for national production projects and a small number of foreign projects. Moreover, official documents are often presented in a rough and less suitable way as a learning resource for beginners.

If you can't read official web files, you can also search for high-quality blogs or videos. **You must not have come up with a way to understand this technology.**

As we learn about the Spring framework, for example, I suggest that you start not just by studying the theory or source code of the Spring framework, but by actually experiencing IoC (Inversion of Control) and AOP (Aspect-Oriented Programming), using the Spring framework to write some demos or even small projects.

In short, **before studying the principles of this technology, understand how it works.**

This step-by-step learning process can gradually help you build a sense of interest in learning, gain a sense of immediate achievement, and avoid being discouraged from directly studying the rationale.

**When studying a certain technical rationale, we can do so in order to avoid being too abstract.**

When we learned Tomcat, for example, we found Tomcat's self-defined thread pool interesting, so we could write a custom version of the thread pool ourselves. And, for example, when we learn the Dubbo principles, we can build a simple version of the RPC framework ourselves.

In addition, there are some differences in the techniques that need to be used in learning projects and in interviews.

If you learn a technology for use in a practical project, your focus is on learning about its use, best practices, and the problems it may encounter in its use. Your ultimate goal is that this technology brings real results to the project and that this effect is positive.

If you study a technology for the sole purpose of an interview, then your focus should be on some of the most common issues in this technology for interviews, that is, the eight parts we often mention.
