# Distributed ID Introduction

What's an ID?

In day-to-day development, we need to use IDs for all data in the system. For example, the user ID corresponds to one person, the commodity ID corresponds to one commodity, and the order ID corresponds to one order.

We also have a variety of IDs in real life, such as ID cards, which correspond to one person, address IDs, and only one address.

In short, **ID is the only identifier for the data.**

What's a distributed ID?

A distributed ID is an ID under a distributed system. Distributed IDs do not exist in real life and are a concept in computer systems.

I will give a brief example of a breakdown.

A project in my division was using a single machine MySQL. However, it was not expected that, one month after the project went online, the amount of data for the system as a whole would increase as more people were using it. The single machine MySQL could no longer be maintained and required a split list (recommended for Sharding-JDBC).

After the partition, the data is spread over the databases on different servers, and the self-incrementing primary key in the database is no longer the only one generated. **How do we generate a single global primary key for different data nodes?**

This is the time to generate **distributed IDs**.

![Distributed ID Example](https://oss.javaguide.cn/github/javaguide/system-design/distributed-system/id-after-the-sub-table-not-conflict.png)

What are the requirements for distributed IDs?

![Distributed ID Requirements](https://oss.javaguide.cn/github/javaguide/system-design/distributed-system/distributed-id-requirements.png)

As an essential part of the distribution system, distributed IDs are used in many places.

A basic distributed ID requires the following:

- **Global uniqueness**: The global uniqueness of the ID must be met first!
- **High performance**: The distributed ID is generated faster and local resources are consumed less.
- **High availability**: The generation of distributed ID services needs to ensure unlimited availability close to 100%.
- **Easy to use**: Take it easy, use it easily, access fast!

In addition to these, a better distributed ID should ensure that:

- **Safety**: ID does not contain sensitive information.
- **Ordered increment**: If IDs are to be stored in the database, the order of IDs can increase the speed of writing to the database. And, many times, it's also likely that we'll sort it directly through the ID.
- **It has a specific business meaning**: If there is a specific business meaning, the resulting ID can make positioning issues and development more transparent (it can be determined which business is through the ID).
- **Independent deployment**: That is, the distribution system has a single transmitter service dedicated to generating distributed IDs. This allows for the generation of ID services that can be decoupled from business-related services. However, this also raises the issue of increased consumption of network calls. In general, stand-alone launcher services are necessary if there are more scenarios that need to be used for distributed IDs.

# Distributed ID Common Solutions

Database

### Database Primary Key Self-Addition

This is simple and straightforward, and is the only ID generated by a relationship-based primary key.

![Database Primary Key](https://oss.javaguide.cn/github/javaguide/system-design/distributed-system/the-primary-key-of-the-database-increas-automatically.png)

By MySQL, for example, we can just go down this way.

**1. Create a database table.**

```sql
CREATE TABLE `security_id` (
  `id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
  `stub` char(10) NOT NULL DEFAULT '',
  PRIMARY KEY (`id`),
  UNIQUE KEY `stub` (`stub`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

The `stub` field is meaningless and is simply a place where we can insert or modify data. Furthermore, a unique index was created for the `stub` field to ensure its uniqueness.

**2. Insert data through `REPLACE INTO`.**

```java
BEGIN;
REPLACE INTO Sequence_id (stub) VALUES ('stub');
SELECT LAST_INSERT_ID();
COMMIT;
```

Insert data here; we do not use `INSERT INTO`, but rather `REPLACE INTO`, as follows:

- Step 1: Try to insert data into the table.
- Step 2: If the primary key or the unique index field fails to insert due to a duplicate data error, remove the conflicting line containing the duplicate key value from the table and then try again to insert the data into the table.

The advantages and disadvantages of this approach are also evident:

- **Advantages**: Simpler to achieve, ID orderly incremental, small storage consumption space.
- **Disadvantages**: Little co-production support, single-point
